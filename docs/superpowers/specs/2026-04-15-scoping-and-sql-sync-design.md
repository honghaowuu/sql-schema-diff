# Design: Database Scoping & SQL Sync File

**Date:** 2026-04-15  
**Status:** Approved

---

## Overview

Two extensions to `mysql-schema-diff`:

1. **Optional database scoping** — a `--ignore-base-only-dbs` flag that restricts diffing to databases that exist on the check instance, silently ignoring databases that only exist on base.
2. **SQL sync file** — a new `sql_generator.rs` module that produces a `.sql` migration file alongside the Markdown report, containing statements to bring the check instance's schema in line with base.

---

## Feature 1: Optional Database Scoping

### Motivation

When base has more databases than check (e.g. base has `orders` but check does not), the current tool reports those as `OnlyInBase`. In many workflows this is noise — the operator only cares about databases that check already manages. The flag suppresses these entries.

### CLI Change

Add to `config.rs` / `Cli`:

```
--ignore-base-only-dbs    [flag, default: false]
    When set, databases present only in the base instance are silently
    excluded from the diff. Databases present only in check are still
    reported as OnlyInCheck.
```

### Differ Change (`differ.rs`)

`diff()` receives the flag. When `true`, `all_dbs` is built from check-side keys only:

```rust
let all_dbs: BTreeSet<String> = if ignore_base_only {
    check.databases.keys().cloned().collect()
} else {
    // current behavior: union of both sides
    base.databases.keys().chain(check.databases.keys()).cloned().collect()
};
```

No other modules change for this feature.

### Behavior Summary

| Database exists in | `--ignore-base-only-dbs` off (default) | `--ignore-base-only-dbs` on |
|---|---|---|
| Base only | Reported as `OnlyInBase` | Silently skipped |
| Check only | Reported as `OnlyInCheck` | Reported as `OnlyInCheck` |
| Both | Diffed normally | Diffed normally |

## Feature 2: SQL Sync File

### Motivation

The Markdown report shows what differs; the SQL file gives operators ready-to-run (or ready-to-review) statements to close the gap.

### Architecture

Follows the existing pipeline pattern. A new `src/sql_generator.rs` module consumes a `&DiffReport` and produces a SQL string plus a list of warnings:

```
differ.rs  →  DiffReport  →  reporter.rs      →  diff.md
                          →  sql_generator.rs  →  diff.sql  +  stderr warnings
```

### Public API

```rust
// src/sql_generator.rs
pub fn generate(report: &DiffReport) -> (String, Vec<String>)
```

`main.rs` calls this, writes the `.sql` file, and prints warnings to stderr.

### Output File Path

Derived from `--output`:
- If `--output diff.md` → also writes `diff.sql`
- If output defaults to `schema-diff-YYYYMMDD-HHMMSS.md` → also writes `schema-diff-YYYYMMDD-HHMMSS.sql`

### SQL File Structure

Each database section is wrapped with `FOREIGN_KEY_CHECKS = 0` to avoid ordering failures when adding FKs to tables that reference each other, and restored at the end.

```sql
-- Generated by mysql-schema-diff
-- Base:  <base_label>
-- Check: <check_label>
-- Date:  <generated_at>
--
-- Apply this file to the CHECK instance to sync it with BASE.
-- Lines starting with  -- [WARN]  are commented out because the object
-- exists in check but not in base. Review carefully before enabling.
-- DELIMITER directives are included for MySQL CLI compatibility.
-- When executing via programmatic APIs (e.g. sqlx), strip DELIMITER lines.

SET FOREIGN_KEY_CHECKS = 0;

-- ============================================================
-- Database: mydb
-- ============================================================
USE `mydb`;

-- ... statements ...

SET FOREIGN_KEY_CHECKS = 1;
```

### Statement Generation Rules

All DDL uses backtick-quoted identifiers. `MODIFY COLUMN` is used (not `ALTER COLUMN`) for MySQL 5.x compatibility. Check constraints are only emitted when present (the fetcher already returns empty lists on MySQL < 8.0).

#### Tables

| Situation | Generated SQL |
|---|---|
| Table only in base | `CREATE TABLE IF NOT EXISTS` reconstructed from `TableDef` columns, indexes, FKs, checks |
| Table only in check | `-- [WARN] table \`t\` exists in check but not in base` + commented `DROP TABLE IF EXISTS \`t\`;` |
| Table changed | Individual column/index/FK/check diff statements (see below) |

#### Columns

| Situation | Generated SQL |
|---|---|
| Column only in base | `ALTER TABLE \`t\` ADD COLUMN \`col\` <type> [NOT NULL] [DEFAULT x] [extra] [AFTER prev_col];` |
| Column only in check | `-- [WARN]` + commented `ALTER TABLE \`t\` DROP COLUMN \`col\`;` |
| Column changed | `ALTER TABLE \`t\` MODIFY COLUMN \`col\` <new_type> ...;` |

Column position (`AFTER`) uses the base ordinal to preserve ordering where possible.

#### Indexes

Partial indexes (TEXT/BLOB columns) use `(col(n))` syntax from `IndexColumn.sub_part`. Index type (BTREE/HASH) is emitted via `USING <type>` only when the type is HASH (BTREE is the default and omitted to reduce noise).

| Situation | Generated SQL |
|---|---|
| Index only in base | `ALTER TABLE \`t\` ADD [UNIQUE] INDEX \`name\` (\`col\`(n), ...) [USING HASH];` (or `ADD PRIMARY KEY` for `PRIMARY`) |
| Index only in check | `-- [WARN]` + commented `ALTER TABLE \`t\` DROP INDEX \`name\`;` |
| Index changed | Commented drop + `ALTER TABLE \`t\` ADD ...` (MySQL requires drop-then-add for index changes) |

#### Foreign Keys

| Situation | Generated SQL |
|---|---|
| FK only in base | `ALTER TABLE \`t\` ADD CONSTRAINT \`name\` FOREIGN KEY (\`cols\`) REFERENCES \`ref\`(\`cols\`) ON DELETE x ON UPDATE y;` |
| FK only in check | `-- [WARN]` + commented `ALTER TABLE \`t\` DROP FOREIGN KEY \`name\`;` |
| FK changed | Commented drop + ADD |

#### Views

| Situation | Generated SQL |
|---|---|
| View only in base | `CREATE OR REPLACE VIEW \`v\` AS <definition>;` |
| View only in check | `-- [WARN]` + commented `DROP VIEW IF EXISTS \`v\`;` |
| View changed | `CREATE OR REPLACE VIEW \`v\` AS <new_definition>;` |

#### Stored Procedures & Functions

MySQL 5.x has no `CREATE OR REPLACE` for routines. The full `CREATE` statement is reconstructed from all `RoutineDef` fields: `definition` (body), `is_deterministic`, `sql_data_access`, and `security_type`.

| Situation | Generated SQL |
|---|---|
| Routine only in base | `DROP PROCEDURE/FUNCTION IF EXISTS \`name\`;` + `DELIMITER $$` + `CREATE [DEFINER=CURRENT_USER] PROCEDURE/FUNCTION \`name\`(...) [DETERMINISTIC] [<sql_data_access>] SQL SECURITY <security_type> <body>` + `$$` + `DELIMITER ;` |
| Routine only in check | `-- [WARN]` + commented drop |
| Routine changed | Drop IF EXISTS + recreate |

Note: The `DEFINER` clause is not captured in `RoutineDef` and is omitted (defaults to `CURRENT_USER` at execution time). This is intentional to avoid privilege issues when applying to a different instance.

#### Triggers

| Situation | Generated SQL |
|---|---|
| Trigger only in base | `DROP TRIGGER IF EXISTS \`name\`;` + `DELIMITER $$` + `CREATE TRIGGER ...` + `$$` + `DELIMITER ;` |
| Trigger only in check | `-- [WARN]` + commented drop |
| Trigger changed | Drop IF EXISTS + recreate |

### Warning Output (stderr)

For each `-- [WARN]` comment emitted in the SQL, `main.rs` also prints to stderr:

```
[WARN] mydb.users: column `old_col` exists in check but not in base — review commented DROP in sync SQL
```

### MySQL 5.x Compatibility

- No `CHECK` constraints emitted (fetcher returns empty list on 5.x)
- `MODIFY COLUMN` used, not `ALTER COLUMN`
- `CREATE OR REPLACE VIEW` available since MySQL 5.0
- Routines and triggers use `DROP IF EXISTS` + `CREATE` pattern (no `CREATE OR REPLACE`)
- `FOREIGN_KEY_CHECKS` wrapping is compatible with all MySQL versions

### Known Limitations

- **Table charset/collation**: `TableDef` does not capture `TABLE_COLLATION`. `CREATE TABLE` statements omit charset/collation and inherit the database default. This is a known gap — extending `TableDef` is left for a future iteration.
- **Generated columns**: MySQL 5.7+ `GENERATED ALWAYS AS` syntax is not captured in `ColumnDef.extra`. Generated columns in base-only tables may not reconstruct correctly. A `-- [WARN]` comment is emitted when `extra` contains `VIRTUAL GENERATED` or `STORED GENERATED`.
- **AUTO_INCREMENT starting value**: The `auto_increment` keyword is preserved from `ColumnDef.extra`, but the starting counter value (from `information_schema.TABLES.AUTO_INCREMENT`) is not fetched and will default to 1 in `CREATE TABLE` statements.
- **View/routine dependency ordering**: Views or routines that reference other views/routines not yet created will fail at apply time. The generated SQL does not topologically sort these; operators must review and reorder if needed.
- **Circular foreign keys**: Although `FOREIGN_KEY_CHECKS = 0` is set, tables with circular FK references may still require manual intervention depending on MySQL version behaviour.
- **DELIMITER directives**: Included for `mysql` CLI compatibility. Strip them when executing via programmatic APIs.
